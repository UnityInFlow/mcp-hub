**Gathered:** 2026-04-08

<domain>
## Phase Boundary

Generate ready-to-paste Claude Code and Cursor config snippets from TOML config (offline) or live introspected state via daemon socket. Pure CLI feature — new `gen-config` subcommand, no UI changes, no new crates.

Requirements: GEN-01, GEN-02, GEN-03, GEN-04, GEN-05.

</domain>

<decisions>
## Implementation Decisions

### Output format & structure
- **D-01:** Minimal JSON output — only `command`, `args`, `env` per server. Matches what Claude Code actually needs in `settings.json`. No extra metadata fields in the JSON.
- **D-02:** Output to stdout only. No `--output` flag. Unix philosophy — users pipe, redirect, or copy-paste. `mcp-hub gen-config --format claude | pbcopy`.
- **D-03:** `//` comment header above the JSON with mcp-hub version and ISO timestamp. Not valid JSON but immediately informative. Example: `// Generated by mcp-hub v0.1.0 at 2026-04-08T14:30:00Z`.
- **D-04:** Cursor config format to be researched by downstream researcher. Exact schema, file location (`.cursor/mcp.json` vs `.cursorrc`), and field differences are TBD. Do NOT assume same as Claude Code.

### Live mode behavior
- **D-05:** `--live` mode adds tool names as `//` comments per server. Example: `// filesystem: tools=[read, write, list, stat]`. Offline mode has no per-server comments.
- **D-06:** Down/unhealthy servers are still included in `--live` output (config from TOML), with a `// WARNING: <name> is not running (state: <state>)` comment. User decides whether to keep it.
- **D-07:** `--live` connects to the running daemon via Unix socket (same IPC mechanism as `mcp-hub status` from Phase 3). If daemon is not running, exit with clear error: "No daemon running. Start with: `mcp-hub start --daemon`".

### Edge cases & warnings
- **D-08:** Env var values passed through as-is — no secret detection, no redaction. User's responsibility to manage secrets in their TOML config.
- **D-09:** `env_file` references are resolved and merged (same as `resolve_env` in config.rs). Output contains the final merged env values. Config snippet is ready to use without the `.env` file.
- **D-10:** Zero servers in config: print warning to stderr (`⚠ No servers configured in mcp-hub.toml`), output valid but empty `{ "mcpServers": {} }` with header comment. Exit 0.

### CLI flag design
- **D-11:** Subcommand name: `gen-config`. Matches spec and requirements. Short, clear.
- **D-12:** `--format` flag is required. Values: `claude`, `cursor`. Error with usage hint if omitted.
- **D-13:** `--live` flag (optional). When present, connects to daemon for introspected state. When absent, generates from TOML config only (offline, no daemon needed).

### Claude's Discretion
- Exact JSON pretty-printing style (indent width, trailing newlines)
- Whether to sort servers alphabetically or preserve TOML order
- Error message wording for missing daemon, bad config, etc.
- Internal module structure for gen-config code
- How to handle `http` transport servers (v1 only supports stdio — include or warn?)

</decisions>

<specifics>
## Specific Ideas

- Output should feel like a "paste this into your settings.json" snippet — minimal, clean, immediately usable
- `//` comments make the output self-documenting without breaking the config (most JSON parsers in editors tolerate `//`)
- `--live` tool comments give users a quick audit of what each server provides without opening the web UI
- Warning comments for down servers in `--live` mode let the user make the call, rather than silently excluding or hard-failing

</specifics>
