## User Constraints (from CONTEXT.md)

### Locked Decisions

**Output format & structure**
- D-01: Minimal JSON output — only `command`, `args`, `env` per server. Matches what Claude Code actually needs in `settings.json`. No extra metadata fields in the JSON.
- D-02: Output to stdout only. No `--output` flag. Unix philosophy — users pipe, redirect, or copy-paste.
- D-03: `//` comment header above the JSON with mcp-hub version and ISO timestamp. Not valid JSON but immediately informative. Example: `// Generated by mcp-hub v0.1.0 at 2026-04-08T14:30:00Z`
- D-04: Cursor config format to be researched by downstream researcher. Exact schema, file location, and field differences are TBD. Do NOT assume same as Claude Code.

**Live mode behavior**
- D-05: `--live` mode adds tool names as `//` comments per server. Example: `// filesystem: tools=[read, write, list, stat]`. Offline mode has no per-server comments.
- D-06: Down/unhealthy servers are still included in `--live` output (config from TOML), with a `// WARNING: <name> is not running (state: <state>)` comment. User decides whether to keep it.
- D-07: `--live` connects to the running daemon via Unix socket (same IPC mechanism as `mcp-hub status`). If daemon is not running, exit with clear error: "No daemon running. Start with: `mcp-hub start --daemon`"

**Edge cases & warnings**
- D-08: Env var values passed through as-is — no secret detection, no redaction.
- D-09: `env_file` references are resolved and merged (same as `resolve_env` in config.rs). Output contains the final merged env values.
- D-10: Zero servers in config: print warning to stderr (`⚠ No servers configured in mcp-hub.toml`), output valid but empty `{ "mcpServers": {} }` with header comment. Exit 0.

**CLI flag design**
- D-11: Subcommand name: `gen-config`.
- D-12: `--format` flag is required. Values: `claude`, `cursor`. Error with usage hint if omitted.
- D-13: `--live` flag (optional). When present, connects to daemon for introspected state.

### Claude's Discretion
- Exact JSON pretty-printing style (indent width, trailing newlines)
- Whether to sort servers alphabetically or preserve TOML order
- Error message wording for missing daemon, bad config, etc.
- Internal module structure for gen-config code
- How to handle `http` transport servers (v1 only supports stdio — include or warn?)

None — discussion stayed within phase scope

## Summary

Phase 5 is the leanest phase in the milestone. It adds a single `gen-config` subcommand that reads from the already-built config layer (Phase 1) and, optionally, queries the already-built daemon IPC layer (Phase 3). There is no new async infrastructure, no new crates for JSON output (`serde_json` is already a dependency), and no new network stack.

The two target formats (Claude Code and Cursor) were verified against official sources. Both use an identical `mcpServers` JSON shape with `command`, `args`, and `env` fields — the only difference is where the user pastes the output: into `~/.claude.json` / `.mcp.json` for Claude Code, or into `~/.cursor/mcp.json` / `.cursor/mcp.json` for Cursor. The generated snippet is the same JSON structure; the two `--format` variants serve as labelled output with a contextual header comment.

The one area requiring a deliberate design choice is timestamp generation: the project has no date/time crate yet. `std::time::SystemTime` is available in Rust's stdlib and produces a seconds-since-epoch value; converting to an ISO-8601 string without a crate requires ~5 lines of formatting code. This is simpler than adding `chrono` as a dependency, and the output is functionally equivalent for the use case (a human-readable header comment).

**Primary recommendation:** Implement `src/gen_config.rs` as a pure module with two public functions (`render_claude_config` and `render_cursor_config`) that accept `&HubConfig` plus an optional `&[ServerLiveInfo]` slice, and return a `String`. Wire it into `cli.rs` (new `GenConfig` variant) and `main.rs` (new match arm). No new async, no new crates required.

## Target Config Schemas (Verified)

### Claude Code `mcpServers` Schema

Claude Code stores MCP servers in `~/.claude.json` (local/user scope) or `.mcp.json` (project scope). [VERIFIED: code.claude.com/docs/en/mcp]

**Exact structure for a stdio server:**

```json
// Generated by mcp-hub v0.1.0 at 2026-04-08T14:30:00Z
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/tmp"],
      "env": {
        "API_KEY": "abc123"
      }
    }
  }
}
```

**Key findings:**
- `command`, `args`, `env` are the standard fields for stdio servers [VERIFIED: code.claude.com/docs/en/mcp]
- A `"type": "stdio"` field appears in HTTP-style entries but is NOT required for stdio servers — Claude Code infers transport from the presence of `command` vs `url` [VERIFIED: code.claude.com/docs/en/mcp]
- `args` and `env` can be omitted if empty, but including them as empty array/object is valid and explicit [VERIFIED: code.claude.com/docs/en/mcp]
- The `//` comment header is not valid JSON but is well-tolerated by copy-paste usage; Claude Code itself does not parse comments from user-generated snippets

### Cursor `mcpServers` Schema

Cursor reads from `~/.cursor/mcp.json` (global) or `.cursor/mcp.json` (project root). [VERIFIED: truefoundry.com, multiple community sources]

**Exact structure for a stdio server:**

```json
// Generated by mcp-hub v0.1.0 at 2026-04-08T14:30:00Z — paste into ~/.cursor/mcp.json or .cursor/mcp.json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/tmp"],
      "env": {
        "API_KEY": "abc123"
      }
    }
  }
}
```

**Key findings:**
- Cursor's schema is **identical** to Claude Code's `mcpServers` shape [VERIFIED: multiple official examples]
- There is **no `"type"` field** in Cursor's stdio server entries; transport is inferred from `command` vs `url` [VERIFIED: truefoundry.com/blog/mcp-servers-in-cursor]
- Cursor supports `~/.cursor/mcp.json` (global) and `.cursor/mcp.json` (project-scoped) [VERIFIED: cursor.com/docs/context/mcp via search result]

**Practical implication for D-04:** The two formats produce structurally identical JSON. The difference is expressed in the header comment: the claude format header says "paste into mcpServers in settings.json or .mcp.json", the cursor format header says "paste into ~/.cursor/mcp.json or .cursor/mcp.json". This is the correct and minimal implementation of D-04.

## Architecture Patterns

### Recommended Module Structure

```
src/
├── gen_config.rs        ← new: pure functions, no async, no I/O
├── cli.rs               ← add GenConfig variant + GenConfigArgs struct
└── main.rs              ← add Commands::GenConfig match arm (async for --live)
```

No subdirectory needed. The entire feature fits in one ~150-line module.

### Pattern 1: Pure Rendering Functions

`src/gen_config.rs` exports two public rendering functions. Both take the same arguments and produce a `String`. They do not perform I/O, do not call the daemon, and do not panic.

```rust
// Source: design derived from existing codebase patterns

/// A minimal representation of per-server live state for --live mode.
/// Derived from DaemonResponse::Status data after IPC call.
pub struct ServerLiveInfo {
    pub name: String,
    pub state: String,            // process_state string from daemon response
    pub tool_names: Vec<String>,  // tool names from capabilities
}

/// Render Claude Code mcpServers JSON.
///
/// `live_info` is None in offline mode; Some slice in --live mode.
pub fn render_claude_config(
    config: &HubConfig,
    live_info: Option<&[ServerLiveInfo]>,
) -> anyhow::Result<String> { ... }

/// Render Cursor mcpServers JSON.
///
/// Same logic as render_claude_config; header comment differs.
pub fn render_cursor_config(
    config: &HubConfig,
    live_info: Option<&[ServerLiveInfo]>,
) -> anyhow::Result<String> { ... }
```

**Why this structure:** Keeping rendering functions pure makes them trivially testable — no mocking, no spawned daemons, no temp files. Tests call `render_claude_config(config, None)` and assert on the returned `String`.

### Pattern 2: Timestamp Without External Crate

```rust
// Source: Rust stdlib SystemTime — [VERIFIED: docs.rust-lang.org]
use std::time::{SystemTime, UNIX_EPOCH};

fn iso_timestamp() -> String {
    let secs = SystemTime::now()
        .duration_since(UNIX_EPOCH)
        .unwrap_or_default()
        .as_secs();
    // Manual ISO-8601 UTC formatting from epoch seconds.
    let s = secs % 60;
    let m = (secs / 60) % 60;
    let h = (secs / 3600) % 24;
    let days = secs / 86400;
    // Simplified date calculation — sufficient for a header comment.
    // For production-quality parsing, add chrono. For a comment header, this is fine.
    format!("{}T{:02}:{:02}:{:02}Z", days_to_date(days), h, m, s)
}
```

**Note:** The above is a sketch. The planner should allocate a task to research whether using `chrono` (already a transitive dep of many crates) or adding it explicitly is preferred. The timestamp only needs to be human-readable in a comment — full ISO-8601 correctness is not load-bearing for this feature.

**Alternative (simpler, no crate):** Output seconds since epoch in the header comment: `// Generated by mcp-hub v0.1.0 at epoch=1744115400`. Less readable but zero code. Not recommended for user-facing output.

**Recommendation:** Add `chrono` dev-dependency (or runtime dependency) for clean `DateTime<Utc>::now()` formatting. It is a ubiquitous crate with no surprises. [ASSUMED — whether `chrono` is acceptable must be confirmed; it is not currently in Cargo.toml]

### Pattern 3: `--live` IPC Call in main.rs

The `--live` path follows the exact same pattern as `Commands::Status` — connect to the daemon socket, send `DaemonRequest::Status`, parse the JSON array response. The difference is that it passes the parsed data as `Some(&live_info)` into the rendering function.

```rust
// Pattern derived from existing Commands::Status arm in main.rs [VERIFIED: src/main.rs]
Commands::GenConfig(args) => {
    let config = config::find_and_load_config(cli.config.as_deref())
        .context("Failed to load config")?;

    let live_info: Option<Vec<ServerLiveInfo>> = if args.live {
        let sock = daemon::socket_path()?;
        let response = control::send_daemon_command(
            &sock,
            &control::DaemonRequest::Status,
            5,
        ).await?;
        Some(parse_live_info_from_response(response)?)
    } else {
        None
    };

    let output = match args.format.as_str() {
        "claude" => gen_config::render_claude_config(&config, live_info.as_deref())?,
        "cursor" => gen_config::render_cursor_config(&config, live_info.as_deref())?,
        _ => anyhow::bail!("Unknown format '{}'. Use --format claude or --format cursor", args.format),
    };

    print!("{output}");
    Ok(())
}
```

### Anti-Patterns to Avoid

- **Async in gen_config.rs:** Keep the module fully synchronous. The IPC call belongs in `main.rs`, not in the rendering logic. Mixing I/O into rendering makes tests fragile.
- **Serializing `ServerConfig` directly with `#[derive(Serialize)]`:** `ServerConfig` includes fields like `transport`, `cwd`, `env_file`, `health_check_interval` that must NOT appear in the output. Build a dedicated output struct or construct `serde_json::Value` manually.
- **Using `unwrap()` on `SystemTime` difference:** Per CLAUDE.md, no `unwrap()` in production. Use `.unwrap_or_default()` for the epoch delta calculation.
- **Printing the empty-servers warning to stdout:** Per D-10, the warning goes to `stderr`. The valid-but-empty JSON goes to `stdout`. Tests must verify both streams separately.

## Common Pitfalls

### Pitfall 1: `HashMap` iteration order gives non-deterministic output

**What goes wrong:** `HubConfig.servers` is `HashMap<String, ServerConfig>`. Iterating it produces servers in a random order on each run. Users notice differing output.

**Why it happens:** `HashMap` does not preserve insertion order.

**How to avoid:** Sort server names alphabetically before rendering. Call `let mut names: Vec<&String> = config.servers.keys().collect(); names.sort();` then iterate `names`. This is Claude's Discretion per D decisions, and alphabetical sort is the right default: deterministic, predictable, testable.

**Warning signs:** Test comparing two gen-config runs fails intermittently.

### Pitfall 2: Serializing `ServerConfig` fields that must not appear in output

**What goes wrong:** Adding `#[derive(Serialize)]` to a custom output struct that accidentally inherits or includes `transport`, `env_file`, `cwd`, `health_check_interval`, `max_retries` — fields that are mcp-hub-internal and have no meaning in Claude Code / Cursor config.

**Why it happens:** Reaching for `serde` auto-derive on the existing `ServerConfig` struct rather than constructing the output value explicitly.

**How to avoid:** Build the output JSON value explicitly:

```rust
// [VERIFIED: pattern derived from serde_json docs]
serde_json::json!({
    "command": server_config.command,
    "args": server_config.args,
    "env": resolved_env,  // from resolve_env(), NOT server_config.env
})
```

Never pass `server_config` directly to a serialize call.

### Pitfall 3: `resolve_env` can fail — must not silently skip it

**What goes wrong:** `config::resolve_env` reads a file from disk (`env_file` path). If the file is missing, it returns `Err`. Ignoring the error and falling back to inline `env` would produce a config snippet with missing credentials.

**Why it happens:** Treating `resolve_env` as a pure function when it has an I/O side effect.

**How to avoid:** Always propagate the error with `?` — let it surface to the user with context: "Cannot read env_file '/path/.env': No such file or directory".

### Pitfall 4: `--live` when daemon is not running produces a confusing socket error

**What goes wrong:** `send_daemon_command` returns an error that mentions the socket path. Users don't know what that means.

**Why it happens:** The underlying error from `tokio::net::UnixStream::connect` is exposed directly.

**How to avoid:** The existing error message in `control::send_daemon_command` already wraps this correctly with "Is the daemon running? Start with: mcp-hub start --daemon" [VERIFIED: src/control.rs:300]. The `main.rs` match arm should propagate this error using `?` without additional wrapping — the existing message is sufficient.

### Pitfall 5: `//` comment as first line breaks JSON parsers

**What goes wrong:** Some tools or scripts that consume the output (`| jq .`) will fail if the `//` header comment appears before the opening `{`.

**Why it happens:** `//` comments are not valid JSON (only JSON5 supports them).

**How to avoid:** This is a known, deliberate trade-off per D-03. The output is meant for copy-paste, not programmatic consumption. Document this clearly. For tooling use cases, the user should strip the header with `tail -n +2`. This pitfall is informational only — do not change the design.

### Full output rendering (offline mode, claude format)

```rust
// Source: design based on existing codebase patterns [VERIFIED: src/config.rs, src/main.rs]
pub fn render_claude_config(
    config: &HubConfig,
    live_info: Option<&[ServerLiveInfo]>,
) -> anyhow::Result<String> {
    let version = env!("CARGO_PKG_VERSION");
    let timestamp = iso_timestamp();

    let mut mcp_servers = serde_json::Map::new();
    let mut names: Vec<&String> = config.servers.keys().collect();
    names.sort();

    for name in names {
        let server = &config.servers[name];
        let resolved_env = config::resolve_env(server)?;

        // Live mode: inject tool comment (handled at string-output level, not JSON level)
        let server_json = serde_json::json!({
            "command": server.command,
            "args": server.args,
            "env": resolved_env,
        });
        mcp_servers.insert(name.clone(), server_json);
    }

    let output_json = serde_json::json!({ "mcpServers": mcp_servers });
    let json_str = serde_json::to_string_pretty(&output_json)?;

    let mut out = format!(
        "// Generated by mcp-hub v{version} at {timestamp}\n{json_str}\n"
    );

    // Inject per-server live comments (insert after each server opening line)
    if let Some(info) = live_info {
        out = inject_live_comments(&out, info);
    }

    Ok(out)
}
```

### Parsing `DaemonResponse::Status` into `ServerLiveInfo`

```rust
// Source: DaemonResponse shape [VERIFIED: src/control.rs]
fn parse_live_info_from_response(
    response: DaemonResponse,
) -> anyhow::Result<Vec<ServerLiveInfo>> {
    anyhow::ensure!(response.ok, "Daemon status query failed: {:?}", response.error);
    let data = response.data
        .ok_or_else(|| anyhow::anyhow!("Empty status response from daemon"))?;
    let servers = data.as_array()
        .ok_or_else(|| anyhow::anyhow!("Expected array from status response"))?;

    servers.iter().map(|s| {
        Ok(ServerLiveInfo {
            name: s["name"].as_str().unwrap_or("").to_string(),
            state: s["state"].as_str().unwrap_or("unknown").to_string(),
            // capabilities.tools is not in current Status response — tools COUNT is.
            // For --live comments, tool names require a richer IPC response (see Open Questions).
            tool_names: vec![], // placeholder
        })
    }).collect()
}
```

## Open Questions

1. **Tool names vs tool count in `--live` mode**

   **What we know:** `DaemonRequest::Status` returns `tools: count` (integer), not tool names. This is what `dispatch_request` in `control.rs` serializes. [VERIFIED: src/control.rs line 204]

   **What's unclear:** D-05 says `// filesystem: tools=[read, write, list, stat]` — individual tool names. The current IPC response only has the count.

   **Options:**
   a. Add a new `DaemonRequest::Capabilities` variant that returns the full `McpCapabilities` struct per server.
   b. Extend `DaemonRequest::Status` to optionally include tool names.
   c. Change D-05 to use count: `// filesystem: tools=4`. Simpler, no IPC change needed.

   **Recommendation:** Option (c) for v1 (use count), with a comment in code noting that a future `DaemonRequest::Capabilities` would enable option (a). The planner should confirm with the user which approach is in scope.

2. **ISO-8601 timestamp crate decision**

   **What we know:** `chrono` is not currently in `Cargo.toml`. `std::time::SystemTime` gives epoch seconds but not calendar dates natively.

   **What's unclear:** Whether adding `chrono` (or `time`) is acceptable per project conventions.

   **Recommendation:** Add `chrono = { version = "0.4", features = ["std"] }` — it is the standard Rust date/time crate, stable, and produces clean `format!("{}", Utc::now().to_rfc3339())` output. [ASSUMED — chrono 0.4.x is current stable; not verified against registry in this session]

3. **`http` transport servers in output**

   **What we know:** `ServerConfig.transport` can be `"http"`. Claude Code and Cursor configs for HTTP servers use `"url"` not `"command"`. [VERIFIED: code.claude.com/docs/en/mcp]

   **What's unclear:** Phase 5 scope says "include or warn?" for http transport servers.

   **Recommendation:** For v1, emit a `// WARNING: <name> uses http transport — manual URL config required` comment and skip the server from the `mcpServers` block. HTTP server configs are fundamentally different (they need a `url` field, not `command`+`args`). Including them as stdio-format entries would produce invalid config.
